## 1. Project Identity

| Field | Value |
|-------|-------|
| **Name** | `agentdiagram` (package.json) |
| **Version** | `0.1.0` |
| **License** | MIT |
| **Runtime** | Next.js 14.2.15 (App Router) on Node.js |
| **Description** | Open-source, locally-runnable diagram-as-code editor and agentic repo explorer inspired by Eraser. |

### Core Value Propositions

1. **No cloud dependency** — the DSL compiler, layout engines, renderer, exporter, and AI pipeline all run locally on the user's machine.
2. **SVG-first rendering** — the same `buildScene()` function drives both the interactive viewport and PNG/SVG exports, guaranteeing pixel-perfect identity.
3. **DSL as source of truth** — all edits (AI-generated, inspector, manual) round-trip through a deterministic text DSL stored in the Monaco editor.
4. **Multi-provider AI** — five AI providers (OpenAI, Anthropic, Gemini, xAI Grok, Azure Foundry) share a single retry-wrapped abstraction; user can switch at any time.
5. **Agentic repo analysis** — a 10-stage pipeline scans any local or GitHub repo and produces an architecture diagram without any manual work.

## 4. Two Operating Modes

AgentDiagram presents a single shell that houses two fundamentally different workflows, both sharing the Monaco editor, diagram canvas, inspector, and export system.

### Mode 1: Code Editor (DSL Render)

The user writes or pastes AgentDiagram DSL text into Monaco. The pipeline is entirely client-side:

```
DSL text (Monaco)
    → Lexer (tokenize)
    → Parser (build AST)
    → Compiler (AST → DiagramIR)
    → Layout Engine (ELK / Graphviz / force-directed)
    → SVG Scene builder (React SVG elements)
    → DiagramCanvas (pan/zoom viewport)
```

No server calls are made during editing. The only server calls are:
- AI fix requests (`POST /api/agent/fix`)
- Export (PDF only, client-side for PNG/SVG)

Inspector edits write directly back to the DSL text using `shared.ts` structured rewrites; positional overrides (drag-move) go to Zustand's `overrides` map.

### Mode 2: Agentic Repo Explorer

The user points the agent at a local path or GitHub URL, selects a provider, optionally sets a focus prompt and diagram type, then triggers the analysis. The pipeline runs server-side over SSE:

```
RepoInput (path / GitHub URL)
    → RepoSource resolution (local | github clone)
    → RepoScanner (fast-glob + allowlist + .gitignore)
    → Classifier (heuristic relevance scoring)
    → DocReader + ImportGraph extraction
    → AnalysisBudget (tier 1-5 based on file count)
    → AnalysisRunner (adaptive parallel LLM summarization)
    → Planner (identifyLayers for multi-layer | generatePlan for single)
    → DSLCompiler (deterministic plan → DSL text)
    → Repair (1 attempt if DSL has syntax errors)
    → validateRenderableDsl (pre-persist validation)
    → [Document Mode] generateTechnicalDocumentation (if enabled)
    → SSE "result" event → client stores DSL + instructionMarkdown → renders
```

Progress is streamed as SSE events (`stage`, `log`, `retry`, `result`, `done`) and displayed via `AnalysisAnimation.tsx`.

#### Document Mode (instructionMode checkbox)

When the **Document Mode** checkbox is ticked in `RepoInput.tsx`, both the Agentic Repo and Multi Layer pipelines append a deep technical documentation pass after diagram compilation. This is powered by `lib/agent/docGenerator.ts` — a dedicated module that is **distinct from** `generateInstructionGuide()` in `customPrompt.ts` (which is used only by the Custom Prompt flow).

**What `docGenerator.ts` provides vs. the old `generateInstructionGuide`:**

| Dimension | Old (`generateInstructionGuide`) | New (`generateTechnicalDocumentation`) |
|---|---|---|
| Context fed to LLM | 7 text lines from plan title/groups/nodes | ALL pre-computed signals (see below) |
| Per-file coverage | None | Every analyzed file with typed exports, surface, side-effects |
| Import graph | Not included | Full graph — who imports what, external package frequencies |
| Route catalog | Not included | Every HTTP route with method, handler path |
| Env var catalog | Not included | Every env var with consuming files |
| Cross-folder boundaries | Not included | Full architectural seam analysis |
| Module rollups | Not included | Aggregated subsystem view (AnalysisDigest) |
| Output style | Generic "build guide" | Authoritative technical reference |
| Large repo strategy | Single call with tiny context | Dual-pass: architecture + module reference |

**Signals fed to the documentation generator:**
- `FileSummary[]` — LLM-extracted per-file: role, category, layer, exports, surface, external_deps, side_effects, notes
- `ImportGraph` — static dependency graph: who imports whom, external package usage frequencies
- `RepoContextDigest` — folder clusters, central hub files (ranked by fan-in/out), API routes, env var usage, export catalogs, cross-folder dependency edges
- `AnalysisDigest` — module rollups with layer assignments, surface aggregation, tier info
- `DocPrior[]` — README and documentation file excerpts (up to 3000 chars each)
- Diagram plan groups / layer catalog — architectural groupings already identified

**Large-repo dual-pass strategy (> 80 analyzed files):**

*Pass 1 — Architecture & Contracts:* covers project overview, tech stack, annotated file tree, system architecture, API endpoint contracts, state management, configuration, security model, testing infrastructure, and module cross-reference map. Uses structural signals (routes, env vars, import graph, folder clusters, module rollups, docs).

*Pass 2 — Module Reference:* covers every analyzed file grouped by directory. For each file: purpose, typed exports, key function signatures with parameters and return types, external dependencies, side effects, and design decisions.

Both passes are concatenated into a single Markdown document separated by `---`. The result is stored as `instructionMarkdown` in the Zustand store and rendered in `InstructionPanel.tsx`.

## 6. Data Flow — End-to-End

### 6.1 DSL Edit → Rendered SVG (client-side only)

```
User types in Monaco
  ↓
Zustand: store.dsl = newText (debounced 200ms)
  ↓
useDerivedIr selector: compile(dsl)
  ↓  returns DiagramIR { nodes[], edges[], groups[], diagnostics[] }
DiagramCanvas receives ir
  ↓
layoutDiagram(ir, overrides) → LayoutResult { positions, sizes }
  ↓  (ELK or force-directed, async, possibly via web worker)
buildScene(ir, layout) → React SVG element tree
  ↓
SVG rendered inside pan/zoom viewport
```

### 6.2 Agentic Repo Analysis → Diagram (server-side, streamed)

```
User fills RepoInput + ProviderConfig + DiagramTypePicker + FocusPromptBox
  ↓
AgentPanel: POST /api/agent/analyze {repoPath, provider, model, kind, focus, quickMode, maxMode}
  ↓
Server: runPipeline(input, sseWriter)
  ↓ SSE stream begins
  [stage: validate]    validateWithRetry(session)
  [stage: scan]        scanRepo(root, allowlist, ignoredFolders) → RepoMap
  [stage: classify]    classifyRelevance(repoMap, kind, focus, topK) → Relevance[]
  [stage: context]     readDocPriors() + extractImportGraph() + buildRepoContext()
  [stage: summarize]   analyzeRelevantFiles() → {budget, assignments, summaries, digest}
  [stage: subsystem]   (heuristic, from layer assignments)
  [stage: plan]        generatePlan(session, planInput) → DiagramPlan
  [stage: compile]     planToDsl(plan) → DSL string
  [stage: validate-dsl] compile(dsl) + tryRepair() if errors
  [stage: instruction] (optional) generateTechnicalDocumentation()
                       ├── single pass  (≤80 files): architecture + module reference in one call
                       └── dual pass   (>80 files): Pass1=architecture/API/config/security/testing
                                                     Pass2=per-file module reference
  [event: result]      {dsl, instructionMarkdown}
  [event: done]
  ↓
Client: AnalysisAnimation parses SSE events → progress UI
  ↓
On "result": store.setDsl(dsl) → triggers client-side render pipeline
```

| `AGENTDIAGRAM_DEFAULT_REPO_PATH` | App | No | Override default scan path (default: `..`) |

### Security Properties

- API keys entered in the UI are held **only in Node.js server-process memory** for the duration of one analysis run. They are never:
  - Written to any disk location
  - Echoed back in any SSE event or HTTP response
  - Logged to stdout
- The repo scanner blocks sensitive system paths via `pathGuard.ts` (`/etc`, `/var`, `~/.ssh`, etc.)
- `.env*`, `*.pem`, `*.key`, `*.crt`, and binary files are excluded from scanning by `repoScanner.ts`
- The file summarization cache (`.agentdiagram-cache/`) stores SHA1-keyed JSON blobs; content is never sent to a third party beyond the provider call it represents
